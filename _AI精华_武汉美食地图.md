# 流量池2026 · 武汉美食地图 精华提炼

---

## 一、项目是什么

一个面向武汉 2026 级大学新生的美食发现网页应用：**"鲜城·味觉地图"**。包含两个独立页面——中南财经政法大学周边版（50 家店）和武汉全城版（540 家店），集成高德地图、分类筛选、搜索排序、收藏导航等功能。本质是一个**引流钩子**，用来吸引大学生关注 Robin 的个人 IP 并加入社群。

---

## 二、目标与诉求

Robin 的真实目标分三层：

1. **表层**：帮大学新生快速找到学校周边和武汉全城的好吃店铺
2. **中层**：通过美食内容吸引粉丝、增加流量池人数（微信群/QQ群），流量池本身可产生广告收入和品牌私域植入
3. **深层**：Robin 的真实 IP 定位是 **"非技术大学生用 AI 做项目（build in public）"**，美食地图只是众多引流钩子中的第一个。最终要沉淀到 AI 主赛道

关键诉求：**先出设计稿，确认后再写代码；页面要有品质感和交互，不能太简陋；免费部署，国内要能正常访问。**

## 三、关键决策与方法

### 数据来源

- **原始数据**：Robin 提供了两份 Excel——一份武汉老店清单（~500 家），一份南湖校区周边店清单（69 家，来自大众点评风格的小吃街数据）
- **坐标获取**：使用高德地图 **Web 服务 API Key**（`c607ef...`），将地址批量转成经纬度（geocode）。前后处理了 502 + 69 = 571 家店的数据
- **校区范围计算**：校区边界外扩 1km（不是从中心点画圆），最终首义校区 8 家 + 南湖校区 42 家 = 财大版共 50 家
- **最终数据**：`campus_data.json`（50 家 + 校园坐标）、`wuhan_all_data.json`（540 家，17 个分类）

### 技术选型

| 决策点 | 选择 | 原因 |
|--------|------|------|
| 地图服务 | 高德地图 JS API | 国内免费、开发者生态最成熟 |
| 页面架构 | 两个独立页面 | 财大版暖橙校园风，全城版深色品质风，分别推 |
| 技术栈 | 纯静态 HTML/CSS/JS | 无需服务器，双击可打开，部署成本最低 |
| 数据存储 | 内联 JSON（转为 JS 变量） | 支持 file:// 直接打开，避免跨域问题 |
| 部署方案 | Netlify + GitHub | Vercel 国内被屏蔽，GitHub Pages 国内慢，Netlify 是国内能访问的免费方案 |
| 自动部署 | GitHub → Netlify 持续部署 | git push 后自动发布 |

### 视觉方向（重大转折）

Robin 最初的设计方案是 AI 在浏览器里画的简陋线稿，被 Robin 直接否决（"太简单太丑了"）。之后重新分工：
- AI 负责数据层和技术方案
- Work Buddy（含 Kimi 模型）负责 UI 设计和前端视觉

但 Work Buddy 只完成了 Ardot 画布上的设计稿，没有写实际代码。最终由 AI 基于设计稿和 README 需求文档，完整实现了前后端。

**最终视觉方案「鲜城·味觉地图」**：统一暖纸基底（#F9F4DF）+ 食欲感色彩系统，两个页面用不同强调色区隔——财大版青柠绿（#65A30D），全城版琥珀金（#D97706）。设计风格受 Airbnb 旅行 App 启发（Soft Airbnb 风格），圆角 16px 卡片、暖奶油底色。

### 重要方向转变

1. **从"1km 半径"到"边界外扩 1km"**：从校区中心点算 1km 只有 2 家店，从校区边界外扩后覆盖大幅增加
2. **从"让 Work Buddy 做前端"到 AI 自己写代码**：Work Buddy 只产出了设计稿，最终前端代码由 AI 实现
3. **从 GitHub Pages 到 Netlify**：因为需要国内用户能无障碍访问
4. **从两个站点到一个**：`stalwart-sherbet` 和 `illustrious-croissant` 双站点导致不同步，最终统一到 `illustrious-croissant`，删除多余站点

## 六、可复用经验

### 值得保留的做法

1. **设计稿先行**：Robin 坚持"先看设计再写代码"，避免了前端做出来不满意再返工。虽然流程上多了设计环节，但整体效率更高
2. **数据与视觉分离**：由 AI 负责数据处理、技术选型、后端支撑；视觉设计交给专门的设计工具/模型。各取所长
3. **纯静态 + 内联数据**：最轻量的 Web 方案，无需服务器、双击可用、任意静态托管都能部署。适合 MVP 和引流类产品
4. **GitHub + Netlify 自动部署**：改完代码 `git push` 就自动上线，零运维成本
5. **分层沟通**：AI 给 Robin 解释技术概念（如 Google Fonts、OG 标签）时用生活类比 + 对比效果，Robin 能快速理解并做决策

### 踩过的坑

1. **高德 API Key 分两种，不能混用**
   - **Web 服务 Key**：用于后端 geocode（地址→坐标）
   - **Web端（JS API）Key**：用于浏览器端渲染地图
   - Robin 一开始只申请了一个，后面发现前端也需要另一个

2. **校区覆盖：从"中心点半径"到"边界外扩"**
   - 从校区中心点算 1km，只覆盖 2 家店
   - 改为从校区边界外扩 1km，覆盖大幅增加到 50 家
   - 数据层面的一个小调整，对产品可用性是质的区别

3. **Work Buddy 只做了设计，没做代码**
   - Robin 以为"做完了"，实际上只是 Ardot 画布上的设计稿
   - 前端页面需要另一个人/工具来实现
   - 后续协作时需明确"设计完成"和"开发完成"的区别

4. **双站点导致的同步问题**
   - 出现了两个 Netlify 站点（`stalwart-sherbet` 和 `illustrious-croissant`），内容不同步
   - Robin 分享出去的链接是旧版，新版没被访问到
   - 最终删除多余站点，统一到一个

5. **GitHub Pages 国内不可用**
   - 默认首选 GitHub Pages，但国内用户访问慢甚至打不开
   - Vercel 也被屏蔽
   - Netlify 是当前可用的免费方案

6. **抖音视频挂链接 = 降流**
   - Robin 实测在抖音视频详情页挂美食地图链接，流量明显下降
   - 抖音算法判定"引导用户离开平台"，直接降权
   - 正确做法：链接只放主页简介，视频内口播引导去主页

7. **Google Fonts 国内不稳定**
   - 导致页面文字延迟显示甚至 fallback 到默认字体
   - 直接移除，改用系统字体，效果几乎无差别但加载更快

8. **域名白名单需要配置**
   - 高德 JS API 的 Key 需要配置域名白名单，否则地图加载不出来
   - 开发阶段可填 `*`，上线后填具体域名
